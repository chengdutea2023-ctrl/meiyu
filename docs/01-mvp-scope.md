# 第一阶段 MVP 范围

## 目标

第一阶段先实现底座统一用户与业务应用接入跑通，让多个独立课件/智能体应用共用一套教师、学生、学校和班级数据。

最新主线：教师和学生在业务底座注册；底座保存平台账号密码、审核教师、维护学生年龄段、学校和班级；第三方业务应用通过 SSO 登录获取当前用户，并通过服务端只读接口读取授权范围内的平台用户。

## 范围

### 用户账号

- email 作为全平台唯一用户标识
- 底座生成统一 `platformUserId`
- 底座保存平台用户密码
- 用户身份：学生、教师、管理员
- 用户审核状态：已通过、待审核、已拒绝
- 学生注册后默认可用
- 教师注册后默认待审核，管理员审核通过后可用
- 用户状态：启用、禁用
- 当前用户上下文 API

### 注册和登录

- 底座提供学生注册
- 底座提供教师入驻申请
- 底座提供统一登录 SSO
- 第三方业务应用跳转到底座登录
- 第三方业务应用不保存平台用户密码
- 第三方业务应用不注册教师/学生

### 业务应用接入

- 管理员登记业务应用
- 每个业务应用拥有 `appId` 和 `appSecret`
- 配置业务应用名称、入口地址、允许调用域名、SSO 回调地址
- 配置业务应用可读取的学校/机构和班级范围
- 业务应用服务端凭 `appId/appSecret` 调用只读 API
- 旧用户同步接口保留路径但禁用

### 基础教育模型

- 机构 / 学校
- 班级
- 平台用户
- 角色
- 用户所属机构
- 用户所属班级
- 业务应用授权范围

### 管理后台

- 管理员登录
- 用户管理
- 教师审核
- 机构管理
- 班级管理
- 业务应用管理
- 应用可读学校/班级范围配置

## 非目标

第一阶段暂不实现：

- 微信支付
- 支付宝支付
- 订单中心
- 权益中心
- 复杂插件市场
- 微信登录
- 手机号登录
- 第三方 OAuth 登录
- 复杂数据看板

## 数据库选择

第一阶段主数据库使用 PostgreSQL。

原因：

- 适合用户、组织、班级、角色、业务应用等强关系数据
- 支持事务、唯一约束、外键和复杂查询
- 适合后续扩展订单、权益、审计日志等严肃业务数据

Redis 只用于缓存和短期状态，例如：

- 应用访问限流
- refresh token 状态
- 验证码
- 临时会话状态
- SSO 一次性授权 code
