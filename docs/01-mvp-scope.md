# 第一阶段 MVP 范围

## 目标

第一阶段先实现业务应用接入和用户同步跑通，让多个独立业务应用可以共用一套智美教育新生态业务底座用户索引。

业务应用有自己的服务器和入口，也保留自己的注册、登录和密码保存。业务应用不直接读取底座数据库，而是在注册成功、登录成功或用户资料变化后，通过标准 API 把用户同步到底座。

## 范围

### 用户账号 / 用户索引

- email 作为全平台唯一用户标识
- 底座生成统一 `platformUserId`
- 底座记录业务应用本地用户和平台用户的绑定关系
- 业务用户密码保存在业务应用，不保存在底座
- 平台管理员账号可以由底座保存密码，用于后台管理
- 用户状态：启用、禁用
- 当前用户信息 API

### 业务应用用户同步

- 业务应用自己注册用户
- 业务应用自己保存密码
- 业务应用自己校验登录
- 登录或注册成功后，业务应用服务端携带 `appId + appSecret` 调用底座
- 底座按 email 创建或归并平台用户
- 底座返回 `platformUserId`
- 业务应用保存 `platformUserId`

### 业务应用接入

- 管理员登记业务应用
- 每个业务应用拥有 `appId` 和 `appSecret`
- 配置业务应用名称、入口地址、允许调用的域名
- 校验业务应用服务端凭证
- 提供业务应用接入文档

### 基础教育模型

- 机构 / 学校
- 班级
- 平台用户
- 角色
- 用户所属机构
- 用户所属班级

### 管理后台

- 管理员登录
- 用户管理
- 机构管理
- 班级管理
- 业务应用管理

### 可选保留能力

- 统一登录页
- SSO 授权 code
- code 换 token

这些能力可以保留给后续需要使用底座登录页的业务应用，但不作为当前主线。

## 非目标

第一阶段暂不实现：

- 微信支付
- 支付宝支付
- 订单中心
- 权益中心
- 复杂插件市场
- 微信登录
- 手机号登录
- 第三方 OAuth 登录
- 复杂数据看板

## 数据库选择

第一阶段主数据库使用 PostgreSQL。

原因：

- 适合用户、组织、班级、角色、业务应用等强关系数据
- 支持事务、唯一约束、外键和复杂查询
- 支持 JSON 字段，方便保留部分扩展配置
- 适合后续扩展订单、权益、审计日志等严肃业务数据

Redis 只用于缓存和短期状态，例如：

- 应用访问限流
- refresh token 状态
- 验证码
- 临时会话状态
- 后续可选 SSO 的一次性登录 code
