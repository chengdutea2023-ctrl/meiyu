# 总体架构草案

## 架构定位

本项目不是单纯的用户系统，而是智美教育新生态业务底座。

底座负责统一账号、统一登录、组织班级、业务应用接入、统一 UI 规范和后续统一支付。业务应用独立开发、独立部署、独立入口，通过底座 API 对接公共能力。

## 第一阶段组件

```text
用户浏览器
  ├─ 访问业务应用 A
  ├─ 跳转到底座登录页
  └─ 登录后回跳业务应用 A

业务应用 A
  ├─ 独立服务器
  ├─ 独立入口
  ├─ 使用 appId / appSecret 接入底座
  └─ 调用底座 API 获取当前用户上下文

智美教育新生态业务底座
  ├─ 统一登录
  ├─ 用户管理
  ├─ 组织 / 班级管理
  ├─ 业务应用管理
  ├─ token 签发与校验
  └─ 管理后台 API

PostgreSQL
  └─ 存储核心业务数据

Redis
  └─ 存储短期状态和缓存
```

## 登录接入流程

```text
1. 用户访问业务应用
2. 业务应用发现用户未登录
3. 跳转到底座浏览器授权入口 `/sso/authorize`
4. 用户在底座登录
5. 底座生成一次性 code
6. 底座回跳业务应用 redirectUri
7. 业务应用服务端用 code + appSecret 换 token
8. 业务应用调用底座 API 获取当前用户
9. 业务应用建立自己的登录态
```

## 后续支付中心方向

支付不进入第一阶段 MVP，但架构上预留统一支付中心。

后续所有业务应用不得直接接微信支付或支付宝，而是通过底座支付中心创建订单、查询订单、接收支付结果通知。

支付中心后续负责：

- 微信支付
- 支付宝支付
- 统一订单号
- 支付回调验签
- 订单状态机
- 退款
- 对账
- 权益发放
- 业务应用 webhook 通知
