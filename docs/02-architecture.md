# 总体架构草案

## 架构定位

本项目不是单纯的用户系统，而是智美教育新生态业务底座。

底座负责教师/学生账号、统一登录、业务应用授权、组织班级、统一 UI 规范和后续统一支付。业务应用独立开发、独立部署、独立入口，通过底座 API 对接公共能力。

当前主线是：用户在业务底座注册和登录；第三方业务应用通过 SSO 获得当前用户，通过服务端只读 API 拉取授权范围内的用户、学校和班级数据。第三方业务应用不再注册平台用户，也不再同步创建平台用户。

## 第一阶段组件

```text
用户浏览器
  ├─ 访问业务应用 A
  ├─ 跳转到底座 SSO 登录
  ├─ 必要时在底座注册学生或提交教师申请
  └─ 登录后回到业务应用 A

业务应用 A
  ├─ 独立服务器
  ├─ 独立入口
  ├─ 独立业务数据库
  ├─ 使用 appId / appSecret 接入底座
  ├─ 使用 code 换 token
  ├─ 调用 /auth/me 获取当前用户上下文
  ├─ 可调用 /app-auth/users 读取授权范围内用户
  └─ 用 platformUserId 关联自己的业务数据

智美教育新生态业务底座
  ├─ 学生注册
  ├─ 教师注册和审核
  ├─ 平台账号密码
  ├─ 统一登录 SSO
  ├─ 学校 / 班级管理
  ├─ 业务应用管理
  ├─ 应用可读范围配置
  └─ 管理后台 API

PostgreSQL
  └─ 存储核心业务数据

Redis
  └─ 存储短期状态和缓存
```

## 用户进入第三方应用流程

```text
1. 用户访问业务应用 A。
2. 业务应用 A 跳转到 /sso/authorize。
3. 用户在底座登录；没有账号时在底座注册。
4. 学生注册后可直接登录；教师注册后等待管理员审核。
5. 底座登录成功后带 code 跳回业务应用 A。
6. 业务应用 A 服务端使用 code + appSecret 换 token。
7. 业务应用 A 调用 /auth/me 获取 user、organizations、classes。
8. 业务应用 A 保存 platformUserId 到自己的 session 或业务表。
9. 业务应用 A 保存自己的业务数据，例如练习记录。
```

## 业务应用读取平台用户流程

```text
1. 管理员在底座后台给业务应用配置可读学校/班级范围。
2. 业务应用服务端携带 X-App-Id / X-App-Secret。
3. 调用 GET /api/v1/app-auth/users。
4. 底座只返回该应用授权范围内、已审核、启用状态的用户。
5. 业务应用根据 platformUserId 关联自己的业务数据。
```

## 用户唯一规则

```text
email 是全平台唯一身份。

底座用户：
  email = student@example.com
  platformUserId = u_abc123

A 应用业务数据：
  platformUserId = u_abc123
  practiceRecord = ...

B 应用业务数据：
  platformUserId = u_abc123
  drawingRecord = ...
```

同一个 email 对应同一个底座用户。第三方应用只保存 `platformUserId`，不保存平台密码。

## 旧同步接口位置

旧的 `POST /api/v1/app-auth/users/sync` 路径仍保留，但已禁用并返回 `403`。

禁用原因：用户注册、密码和身份归属必须回到底座统一管理，避免多个第三方应用各自保存密码、各自注册用户，造成账号体系混乱。

## 后续支付中心方向

支付不进入第一阶段 MVP，但架构上预留统一支付中心。

后续所有业务应用不得直接接微信支付或支付宝，而是通过底座支付中心创建订单、查询订单、接收支付结果通知。

支付中心后续负责：

- 微信支付
- 支付宝支付
- 统一订单号
- 支付回调验签
- 订单状态机
- 退款
- 对账
- 权益发放
- 业务应用 webhook 通知
