# 总体架构草案

## 架构定位

本项目不是单纯的用户系统，而是智美教育新生态业务底座。

底座负责统一用户索引、业务应用授权、组织班级、业务应用接入、统一 UI 规范和后续统一支付。业务应用独立开发、独立部署、独立入口，通过底座 API 对接公共能力。

当前主线不是让所有用户到业务底座统一注册和统一登录，而是让业务应用保留自己的注册和登录入口。业务应用自己保存密码并校验登录，登录成功后通过 API 把用户同步到底座。底座使用 email 作为全平台唯一身份，并生成统一 `platformUserId`。

## 第一阶段组件

```text
用户浏览器
  ├─ 访问业务应用 A
  ├─ 在业务应用 A 注册 / 登录
  └─ 使用业务应用 A 的登录态

业务应用 A
  ├─ 独立服务器
  ├─ 独立入口
  ├─ 自己保存用户密码
  ├─ 自己校验用户登录
  ├─ 使用 appId / appSecret 接入底座
  ├─ 调用底座 API 同步用户
  └─ 保存底座返回的 platformUserId

智美教育新生态业务底座
  ├─ 用户统一索引
  ├─ 应用用户绑定关系
  ├─ 管理员账号登录
  ├─ 组织 / 班级管理
  ├─ 业务应用管理
  ├─ 应用凭证校验
  └─ 管理后台 API

PostgreSQL
  └─ 存储核心业务数据

Redis
  └─ 存储短期状态和缓存
```

## 业务应用用户同步流程

```text
1. 用户访问业务应用 A
2. 用户在业务应用 A 注册或登录
3. 业务应用 A 在自己的数据库校验 email + password
4. 校验成功后，业务应用 A 服务端携带 appId / appSecret 调用底座
5. 底座按 email 查找或创建平台用户
6. 底座生成或返回 platformUserId
7. 底座记录 app_a / externalUserId -> platformUserId 的绑定关系
8. 业务应用 A 保存 platformUserId
9. 业务应用 A 使用自己的 session 维持登录态
```

## 用户归并规则

```text
email 是全平台唯一身份。

A 应用同步 teacher@example.com：
  app_a / a_10001 -> platformUserId u_abc123

B 应用同步 teacher@example.com：
  app_b / b_20001 -> platformUserId u_abc123
```

同一个 email 在不同业务应用中应归并为同一个平台用户。业务应用必须保证自己同步的 email 来源可信，最好已经完成邮箱验证。

## 统一登录 SSO 位置

统一登录 SSO 可以保留为可选能力，适合后续希望使用底座登录页的业务应用。

但第一阶段主线以“业务应用自有注册登录 + 底座用户同步 API”为准。

## 后续支付中心方向

支付不进入第一阶段 MVP，但架构上预留统一支付中心。

后续所有业务应用不得直接接微信支付或支付宝，而是通过底座支付中心创建订单、查询订单、接收支付结果通知。

支付中心后续负责：

- 微信支付
- 支付宝支付
- 统一订单号
- 支付回调验签
- 订单状态机
- 退款
- 对账
- 权益发放
- 业务应用 webhook 通知
